# God Code v0.8: Visual Intelligence + Platform Auth

**Date**: 2026-04-05
**Goal**: Add vision-driven UI iteration loop, platform API key auth for quality testing, performance cleanup, and live Godot runtime bridge.
**Execution order**: WS1 → WS2 → WS3 → WS4 (sequential)
**Prerequisite**: v0.7.0 merged (done: `b7a7fef`)

---

## Executive Summary

v0.7 made god-code genre-aware with 11 skills, sprite QA, polish rubric, and 14 scenario specs. v0.8 closes the remaining gaps:

1. **Performance tax** — reviewer/quality_gate duplicate checks, pixel-loop image comparison, monolithic CLI
2. **No visual intelligence** — god-code can screenshot but can't understand what it sees or iterate on visual quality
3. **No platform testing** — can't measure whether centralized routing actually improves output quality
4. **No live connection** — runtime evidence is synthetic or headless, never from a running game

## WS1: Performance + Tech Debt

### Problem

- reviewer.py and quality_gate.py independently run the same checks (lint, tscn validate, consistency, resource validate, ui layout, audio nodes) — 2x validation cost per turn
- Pixel-by-pixel image comparison in visual_regression.py and sprite_qa.py — Python loops, slow for large images
- cli.py is 900+ lines mixing commands, menus, engine wiring, and config management
- intent_resolver rglobs the entire project on every intent refresh

### Design

#### 1A. Shared Validation Suite

```python
# godot_agent/runtime/validation_checks.py (new)
class ValidationSuite:
    """Run all checks once, share results between quality_gate and reviewer."""

    def __init__(self, project_root, changed_files):
        self._results: dict[str, CheckResult] = {}

    async def run_all(self) -> dict[str, CheckResult]:
        """Execute all checks once, cache results."""

    def get(self, check_name) -> CheckResult | None:
        """Retrieve cached result."""
```

- quality_gate.py creates ValidationSuite, runs checks, builds gate report
- reviewer.py receives the same suite instance — no duplicate execution
- engine.py creates suite once in quality_gate phase, passes to reviewer phase

#### 1B. Image Comparison via Pillow ImageChops

Replace pixel-by-pixel Python loops with Pillow C-level operations:

```python
from PIL import ImageChops
diff = ImageChops.difference(img1, img2)
extrema = diff.getextrema()    # per-channel (min, max)
bbox = diff.getbbox()          # bounding box of differences
```

No numpy dependency — Pillow's ImageChops is C-implemented and sufficient.

#### 1C. CLI Split

```
godot_agent/cli/
├── __init__.py        # Click group + main entry
├── commands.py        # chat, ask, status, setup, mcp
├── menus.py           # provider/model/effort/skills/settings menus
├── engine_wiring.py   # build_engine, build_registry
└── helpers.py         # toolbar, project details, session save/load
```

#### 1D. Intent Resolver Token Cache

```python
class IntentResolver:

    _cache_project_root: str = ""

    def _scan_project_tokens(self, project_root):

            return self._cached_tokens
```

~400 lines changed, 0 new dependencies. All existing 399 tests must pass.

## WS2: Visual Iteration Loop

### Problem

god-code can screenshot, analyze code, and modify scenes — but lacks a closed loop: see screenshot → judge issues → change code → re-screenshot → confirm improvement. Screenshots are currently only used for visual regression (baseline comparison), not for understanding content and iterating.

### Design

#### Core Flow

```
User: "improve this battle scene's UI"
           │
           ▼
    ┌──────────────────┐
    │ 1. Screenshot     │  headless or live bridge
    └────────┬─────────┘
             ▼
    ┌──────────────────────────┐
    │ 2. GPT-5.4 vision        │  via god-code-api (worker role)
    │    → structured           │  outputs: [{node_path, property,
    │      improvement plan     │            old_value, new_value, reason}]
    └────────┬─────────────────┘
             ▼
    ┌──────────────────────────┐
    │ 3. Execute changes        │  scene_tools / script_tools
    └────────┬─────────────────┘
             ▼
    ┌──────────────────────────┐
    │ 4. Re-screenshot          │
    └────────┬─────────────────┘
             ▼
    ┌──────────────────────────┐
    │ 5. Gemini 3 Flash score   │  via god-code-api (playtest_analyst role)
    │    → before/after compare │  5 dimensions, 1-5 scale
    └────────┬─────────────────┘
             ▼
        score >= 4.0?
        ├── yes → done, record baseline
        └── no  → back to step 2 (max 3 rounds)
```

#### New Components

**AnalyzeScreenshotTool** (`godot_agent/tools/vision_analysis.py`):

```python
class AnalyzeScreenshotTool(BaseTool):
    class Input(BaseModel):
        screenshot_path: str = ""      # existing path, or empty = auto-capture
        scene_path: str = ""           # scene to screenshot
        focus: str = "general"         # general | ui | gameplay | visual_quality
        project_path: str = "."

    class Output(BaseModel):
        analysis: str
        suggestions: list[dict]        # [{node_path, property, old_value, new_value, reason}]
        score: float
        provider: str
```

Sends screenshot base64 + analysis prompt to vision model via god-code-api orchestrate endpoint (worker role → GPT-5.4).

**ScoreScreenshotTool** (`godot_agent/tools/vision_scoring.py`):

```python
class ScoreScreenshotTool(BaseTool):
    class Input(BaseModel):
        screenshot_path: str
        baseline_path: str = ""
        criteria: str = "demo_quality"
        project_path: str = "."

    class Output(BaseModel):
        scores: dict[str, float]       # {ui_layout: 4.2, readability: 3.8, ...}
        overall: float
        improved: bool
        feedback: str
```

Sends to Gemini 3 Flash via god-code-api (playtest_analyst role).

**Vision Prompt Templates** (`godot_agent/prompts/vision_templates.py`):

- `ANALYSIS_PROMPT` — full game screenshot analysis (tested: 26/30 score)
- `ITERATION_PROMPT` — before/after comparison with severity ratings
- `SCORING_PROMPT` — 5-dimension JSON output scoring

**Engine Phase**: `RUN_VISUAL_ITERATION`

- Triggers when: user requests UI improvement AND scene has headless screenshot capability
- Controls: `max_visual_iterations: 3` (configurable)
- Records: before/after screenshots + scores in `.god-code-artifacts/visual_iterations/`

#### god-code-api: Gemini Native Adapter

Update `providers/gemini.ts` to support `generateContent` native API with `inline_data` for image payloads. Detect image in request → use native endpoint; text-only → keep OpenAI-compatible.

### Estimate

~500 lines (god-code) + ~80 lines (god-code-api gemini adapter).

**Create:**

## WS3: god-code-api API Key Auth + Platform Keys

### Problem

god-code-api is currently BYOK only. To test whether centralized routing improves quality, the backend needs its own provider keys, an auth mechanism, and usage tracking.

### Design

#### Auth Flow

```
god-code request
    │
    ▼
  Has Authorization: Bearer gc_live_xxx?
    ├── yes → lookup D1 api_keys → use platform provider keys
    └── no  → check request body for provider_keys → BYOK (existing)
```

#### API Key Schema (D1)

```sql
-- schema/003_api_keys.sql
CREATE TABLE api_keys (
  id TEXT PRIMARY KEY,
  user_id TEXT NOT NULL,
  key_hash TEXT NOT NULL UNIQUE,
  label TEXT DEFAULT '',
  quota_total INTEGER DEFAULT 10000,
  quota_remaining INTEGER DEFAULT 10000,
  quota_reset_at TEXT,
  is_active INTEGER DEFAULT 1,
  created_at TEXT NOT NULL,
  last_used_at TEXT
);

CREATE TABLE usage_log (
  id TEXT PRIMARY KEY,
  api_key_id TEXT NOT NULL,
  timestamp TEXT NOT NULL,
  agent_role TEXT NOT NULL,
  provider TEXT NOT NULL,
  model TEXT NOT NULL,
  prompt_tokens INTEGER,
  completion_tokens INTEGER,
  cost_estimate REAL,
  quality_score REAL
);
```

#### Platform Provider Keys

Stored as Worker secrets (never in code):

```bash
wrangler secret put PLATFORM_OPENAI_KEY
wrangler secret put PLATFORM_ANTHROPIC_KEY
wrangler secret put PLATFORM_GEMINI_KEY
```

#### Key Management

- Admin endpoint: `POST /v1/admin/keys` (protected by admin secret)
- Key format: `gc_live_` + 32 random chars
- Storage: SHA-256 hash only in D1
- Quota: per-key, configurable, monthly reset

#### Usage Tracking

- Every platform-mode request → decrement quota + insert usage_log
- `GET /v1/usage` → returns quota status + cost breakdown by provider + quality averages

#### god-code Client

```python
# config.py
backend_api_key: str = ""  # gc_live_xxx

# client.py — _chat_via_backend
if self.config.backend_api_key:
    headers["Authorization"] = f"Bearer {self.config.backend_api_key}"
    # No provider_keys needed — backend uses platform keys
```

#### OAuth Preparation

Auth layer interface designed so OAuth 2.0 is additive:
- Add `POST /v1/auth/authorize` + `POST /v1/auth/token`
- `authenticateRequest` adds JWT decode branch → same platform key path
- god-code CLI adds `god-code login` (browser → callback → store token)
- Core logic (platform keys, quota, usage) fully reused.

### Estimate

~250 lines (god-code-api) + ~30 lines (god-code client) + D1 migration.

### Files

**Create:**
- `god-code-api/schema/003_api_keys.sql`

**Modify:**
- `god-code-api/src/auth/guard.ts` — platform auth mode
- `god-code-api/src/index.ts` — admin endpoints, usage endpoint
- `god-code-api/wrangler.toml` — new secret bindings
- `godot_agent/runtime/config.py` — backend_api_key field
- `godot_agent/llm/client.py` — bearer token header

## WS4: Live Runtime Bridge

### Problem

`RuntimeSnapshot.source: "live_editor"` is an empty path. No actual Godot plugin exists to connect to a running game. Visual iteration (WS2) and playtests are limited to headless screenshots without game state.

### Design

#### Architecture

```
Godot 4.4 (running)                   god-code
┌───────────────────┐                 ┌──────────────────┐
│ GodCodeBridge.gd  │◄──TCP:9394───► │ LiveRuntimeClient │
│ (autoload plugin) │   JSON-RPC     │ (asyncio TCP)     │
└───────────────────┘                 └──────────────────┘
```

Why TCP: Godot's `StreamPeerTCP` is simplest, `asyncio.open_connection` matches, loopback needs no TLS.

Why not Godot Remote Debugger: undocumented, version-fragile, not customizable.

#### Godot Plugin: GodCodeBridge.gd

Autoload script (~200 lines GDScript) listening on TCP port 9394. JSON-RPC protocol, newline-delimited messages.

**RPC Methods:**

| Method | Purpose | Response |
|--------|---------|----------|
| `ping` | Connection check | `{"pong": true}` |
| `get_scene_tree` | Full node list | `[{path, type, visible, children_count}]` |
| `get_node_properties` | Node properties | `{position, size, text, modulate, ...}` |
| `capture_viewport` | Viewport to base64 PNG | `{"image_b64": "..."}` |
| `inject_action` | Simulate input | `{"ok": true}` |
| `get_signals` | Recent emitted signals | `[{signal_name, source, tick}]` |

#### god-code Client: LiveRuntimeClient

```python
# godot_agent/runtime/live_client.py (new)
class LiveRuntimeClient:
    async def connect(self) -> bool
    async def get_scene_tree(self) -> list[dict]
    async def get_node_properties(self, path: str) -> dict
    async def capture_viewport(self) -> str  # base64 PNG
    async def inject_action(self, action: str, pressed: bool) -> bool
    async def build_snapshot(self) -> RuntimeSnapshot  # source="live_editor", evidence_level="high"
    async def close(self)
```

Auto-reconnect on disconnect. Timeout protection on all calls. Silent fallback to headless if connection fails — never blocks the engine.

#### Engine Integration

In `PREPARE_CONTEXT` phase: try live bridge connect → if success, update snapshot with `source="live_editor"`, `evidence_level="high"`. If fail, continue with headless/synthetic — no error.

#### WS2 Enhancement

With live bridge, visual iteration upgrades:
- Headless: screenshot → analyze → change code → re-screenshot
- Live: live viewport capture → analyze → change code → Godot hot-reload → live capture (sees real animations, particles, runtime state)

#### Installation

```bash
god-code setup-bridge  # new CLI command
# → copies god_code_bridge.gd to project's addons/
# → registers autoload in project.godot
# → prompts user to restart Godot
```

### Estimate

~200 lines GDScript + ~150 lines Python + ~50 lines CLI.

### Files

**Create:**
- `godot_agent/addons/god_code_bridge/god_code_bridge.gd`
- `godot_agent/runtime/live_client.py`

**Modify:**
- `godot_agent/runtime/engine.py` — try live bridge in PREPARE_CONTEXT
- `godot_agent/cli/commands.py` — setup-bridge command

### Success Criteria

1. Quality gate + reviewer run checks once per turn, not twice
2. god-code can screenshot a scene, analyze it with GPT-5.4, make changes, re-screenshot, score with Gemini, and iterate up to 3 rounds automatically
3. god-code can authenticate with `gc_live_xxx` API key, use platform provider keys, and query usage/quality stats
4. god-code can connect to a running Godot game via TCP bridge and capture live viewport screenshots with `evidence_level="high"`

### Non-Goals for v0.8

- Full OAuth 2.0 flow (v0.9)
- Dashboard UI for usage/quality data (CLI queries sufficient)
- 3D game support
- Multi-user billing
