# god-code v1.0.0 — UX Upgrade Design Doc

**Date**: 2026-04-07
**Status**: Draft (awaiting approval)
**Author**: Claude Code session
**Scope**: Comprehensive UX overhaul consolidating what was originally scoped as v0.9.3 + v0.9.4 + v0.10.0 into a single major release that bumps god-code to 1.0.0

---

## 0. TL;DR

god-code has 28 identified UX issues spanning 8 thematic categories. The most painful ones (silent 60-second waits, silently truncated outputs, broken token counts, fake Ctrl+C, silent data loss) make the tool feel broken even when it's working correctly. This design doc proposes v1.0.0 as a coordinated fix for all 28 issues, split into 11 reviewable commits, landing in a single release.

**Semantic versioning rationale**: This is the first release where god-code feels like a tool you could recommend to a colleague without a 5-minute apology about its quirks. That's what 1.0 means. No breaking changes to external API — config fields are additive with defaults, session format unchanged, tool contracts unchanged. The bump is about stability and polish, not feature-completeness.

---

## 1. Goals

1. **No silent failures.** Every error reaches the user via at least one of: activity log, event, panel, or inline message. Zero `except Exception: pass`.
2. **Live feedback on every long operation.** No operation over 2 seconds without a visual indicator (spinner, progress, streaming chunks).
3. **Data integrity transparency.** Nothing gets truncated, dropped, or skipped without the user being told.
4. **Recoverable errors.** Ctrl+C actually cancels. Failed `/resume` doesn't half-mutate state. Setup doesn't overwrite config without confirmation.
5. **Clear visual hierarchy between main turn and sub-agent turns.** Planner output is obviously distinct from the main assistant response.
6. **Discoverable features.** Commands, shortcuts, and modes are findable by a first-time user in under 60 seconds.
7. **Correct token accounting.** Backend-mode users see real usage numbers, not zeros.
8. **Honest mode/role reporting.** The planner agent stops saying "我目前是 PLAN 模式" when the user is in `apply` mode.

## 2. Non-goals

1. **No new features.** No new tools, no new modes, no new integrations, no new providers.
2. **No breaking API changes.** Existing configs, sessions, tool contracts, event kinds all continue to work.
3. **No new runtime dependencies.** Stays within the existing `rich` + `prompt_toolkit` + `httpx` + `pydantic` set.
4. **No new backend endpoints.** god-code-api stays as-is (only `LATEST_VERSION` bumps).
5. **No architectural rewrites beyond what's justified by a specific issue.** Cancellation architecture is the one exception — it's explicitly scoped and bounded.
6. **No rebrand / visual redesign.** Same cyan assistant panels, same layout conventions, just with better content.
7. **No site / landing page changes.** This is a CLI release; `god-code-site` gets a version badge bump only.

## 4. Guiding principles

1. **Evidence over assumption.** Every proposed fix has a file:line reference backing the problem and a specific code change backing the fix. No "I think this might help" changes.
2. **Additive over rewrite.** Where a small change fixes the issue, use it. Only rewrite when the architecture itself is wrong (C2 cancellation, A2 tool progress).
3. **Test before fix.** Every issue gets a regression test before the production change. Red → green. No exceptions.
4. **Small commits, grouped logically.** 11 commits (see §10), each independently reviewable and revertable, each with its own tests passing.
5. **Defense in depth.** For high-impact failures, fix them at multiple layers: event + activity log + panel. Don't rely on one surface to catch every case.
6. **Backward compatibility as the default.** New config fields have defaults. New events are additive. Existing tests continue to pass. Any exception requires explicit justification in the commit message.
7. **UX claims need proof.** Every "the user should see X" is verified by a test that asserts the rendered output, not just the state.

## 5. Architectural decisions

The bulk of the issues are localized patches (change one line, add one handler). This section covers the 5 items that require architectural thought.

### 5.1 Cancellation architecture (fixes C2)

**Problem**: Current chat loop catches `KeyboardInterrupt` synchronously and prints "Cancelled", but `await engine.submit(user_input)` isn't actually cancellable. The HTTP request in flight continues in the event loop, any running tool continues in its subprocess, and by the time the future eventually resolves, the engine has already appended messages for that "cancelled" turn. Next turn has polluted state.

**Evidence**: `cli/commands.py:1724-1732`:
```python
if cfg.streaming and engine.on_stream_chunk:
    response = await engine.submit(user_input)
else:
    with display.thinking():
        response = await engine.submit(user_input)
    display.agent_response(response)
except KeyboardInterrupt:
    display.info("Cancelled")
    continue
```

No `task.cancel()`, no cleanup, no rollback.

**Design**:

Wrap `engine.submit()` in an explicit `asyncio.Task`. On `KeyboardInterrupt`, `cancel()` the task, wait up to 2s for cleanup, then call a new `engine.rollback_current_turn()` method to remove any partial state.

```python
# New chat-loop pattern (commands.py)
submit_task = asyncio.create_task(engine.submit(user_input))
try:
    response = await submit_task
except asyncio.CancelledError:
    display.info("Cancelled")
    engine.rollback_current_turn()
    continue
except KeyboardInterrupt:
    submit_task.cancel()
    try:
        await asyncio.wait_for(submit_task, timeout=2.0)
    except (asyncio.CancelledError, asyncio.TimeoutError):
        pass
    display.info("Cancelled — cleaning up")
    engine.rollback_current_turn()
    continue
```

`KeyboardInterrupt` in asyncio is tricky on Python 3.12 — it can arrive either as a synchronous exception at the `await` point or as a `CancelledError` depending on signal handling. The try/except chain handles both.

**New method**: `ConversationEngine.rollback_current_turn()` removes all messages appended since the last user message, clears `state.pending_response`, `state.modified_files`, and resets any uncommitted turn state. It does NOT undo file modifications — those remain on disk (because they may have been real intermediate writes), but the message history no longer references them, so the next turn starts clean.

**Cancellation-safety audit**:
- `httpx.AsyncClient.stream()` context manager → closes on CancelledError via `__aexit__` (native async support). ✅
- `stream_chat_with_callback` → awaits `resp.aiter_lines()`, which is cancellable at the `async for` boundary. ✅
- Tool execution → `await self.registry.execute(tc.name, args)` — cancellable at the await. Subprocess-based tools (run_godot, screenshot, generate_sprite) get the CancelledError at their internal `await process.wait()`, and `process.terminate()` should be called on cleanup. **Needs per-tool audit**: listed in implementation checklist.
- Quality gate and reviewer passes in `_run_loop` → each is a separate await, so cancellation unwinds naturally.

**Subprocess caveat**: If a tool has already spawned a subprocess and that subprocess is mid-execution (e.g., running a 30-second Godot validation), cancelling the Python task doesn't automatically kill the subprocess. Proper fix: tools that spawn subprocesses register them with an engine-level "active subprocess registry", and `rollback_current_turn()` signals them to terminate. For v1.0.0 scope, we implement this only for the handful of known slow tools (godot_cli.py, screenshot.py, image_gen.py) — everything else relies on asyncio's default cancellation.

**Test strategy**:
- Unit: mock an LLM stream that blocks for 10s, start `engine.submit()` as task, cancel after 100ms, verify task ends within 2s, verify `engine.messages` contains no assistant turn from the cancelled submission, verify `engine.last_turn is None` or unchanged.
- Integration: use a real subprocess tool with a mock that sleeps, cancel mid-execution, verify subprocess is terminated and doesn't linger.

**Risk**: **High**. Async cancellation is the hardest-to-test area in this release. Mitigation: extensive mocking, manual smoke test before tagging, design the new method to be additive (old behavior still works if the new cancellation path fails).

### 5.2 Sub-agent panel rendering (fixes A4)

**Problem**: After v0.9.2, the planner sub-engine streams through the same display callbacks as the main turn. Both produce cyan panels titled "assistant". User sees two consecutive cyan panels per turn and can't tell which is the planner's output and which is the main turn.

**Design considerations**:

Option A — rename panel via new parameter: add `role` param to `agent_streaming_start(role="assistant")`, pick title/color per role. **Downside**: changes callback signatures throughout engine, dispatcher, tests, and display.

Option B — visual section markers via events: emit `planner_started` / `planner_finished` events that the display renders as dim horizontal rules with a label. Planner output still streams through the normal path, but it's visually bracketed.

Option C — entirely separate display methods: `subagent_streaming_start()`, `subagent_streaming_chunk()`, etc. **Downside**: doubles the display API surface.

**Decision**: **Option B**. Minimal code change, no signature churn, works with existing event propagation.

**Implementation**:

In `display.handle_event` (new handlers):
```python
elif event.kind == "planner_started":
    self.console.print()
    self.console.print(Rule("[dim cyan]planner[/]", style="dim cyan"))
    self.add_activity("planner: composing plan")
elif event.kind == "planner_finished":
    self.console.print(Rule(style="dim cyan"))
    self.console.print()
    used = event.data.get("used_tools") or []
    suffix = f" ({', '.join(used[:3])})" if used else ""
    self.add_activity(f"planner: plan ready{suffix}")
```

(Uses `rich.rule.Rule` for the horizontal separator — no new dependency, it's already imported via `rich`.)

Visual result:
```
──── planner ────

[cyan box: planner's streamed output]

──────────────────

[cyan box: main turn's streamed output]
```

Worker sub-agents (if/when wired similarly) get analogous events: `worker_started`, `worker_finished`. Reviewer sub-agents already exist but don't stream — their output comes via `_run_reviewer_for_round` which synthesizes a panel separately.

**Test**: capture the `console.print` calls during a mock planner pass, assert that the expected Rule instances were rendered.

### 5.3 Tool progress feedback (fixes A2)

**Problem**: `tool: <name>` line appears, then nothing until `tool: ok/fail`. For a 30-second `generate_sprite` or a 60-second `run_godot` validation, user sees 30-60 seconds of nothing.

**Design**:

Two-layer approach:

**Layer 1 (universal, v1.0.0)**: Spinner during tool execution. `display.tool_start` opens a `rich.console.status("[dim]<tool> running…[/]", spinner="dots")` context and stores it on the display. `display.tool_result` closes it. This works for every tool with zero per-tool instrumentation.

```python
# display.py
def tool_start(self, name: str, label: str) -> None:
    self.add_activity(f"tool: {name}")
    self.console.print(f"  [yellow]▸[/] {name}")
    if label:
        self.console.print(f"  [dim]> {label[:100]}{'…' if len(label) > 100 else ''}[/]")
    # New: spinner while tool runs
    self._tool_status = self.console.status(
        f"[dim]{name} running…[/]", spinner="dots"
    )
    self._tool_status.start()

def tool_result(self, name: str, success: bool, summary: str) -> None:
    if self._tool_status is not None:
        self._tool_status.stop()
        self._tool_status = None
    # ... existing result rendering
```

**Layer 2 (selective, defer to v1.0.1+)**: Tools that know they're slow emit `tool_progress` events with status updates. This requires per-tool code changes (screenshot.py, image_gen.py, godot_cli.py) and is out of v1.0.0 scope.

**Caveat**: `rich.console.status` and the streaming `Live` widget don't compose well — you can't have both running at once. For tools called during streaming, the spinner is skipped (streaming already shows movement). This is fine because tool calls always happen between streaming phases, never during them.

**Test**: mock a tool that sleeps, call `tool_start` → assert `_tool_status is not None`, call `tool_result` → assert `_tool_status is None`.

### 5.4 Transactional resume (fixes C3)

**Problem**: `/resume` path partially mutates engine state (`_replace_engine` loads messages, reconfigures display, re-wires callbacks) **before** checking whether the saved project path still exists. If validation fails, some of the mutations have already happened. Next turn may execute in the wrong working directory.

**Evidence**: `cli/commands.py:916-919` (approximate, exact lines to verify during implementation).

**Design**: Swap validation order. Validate **before** mutating.

```python
# Before (buggy):
_replace_engine(saved_path, ...)  # mutates state
if not Path(saved_project).exists():
    display.error("Saved project path no longer exists")
    return

# After:
saved_project = _read_session_metadata(session_id)["project_path"]
if not Path(saved_project).exists():
    display.error(f"Saved project path no longer exists: {saved_project}")
    display.info("Use /cd <valid_path> first, or /new to start fresh")
    return
_replace_engine(saved_path, ...)
```

The fix is simple in theory but requires a careful read of `_replace_engine` to make sure there are no other silent mutations hidden inside.

**Test**: mock a session file whose project path doesn't exist, call `/resume <id>`, assert engine state is unchanged (messages, project_path, etc.).

### 5.5 Planner prompt redesign (fixes F1, F2)

**Problem 1 (F1)**: Planner agent's system prompt contains language like "You are in plan mode" or "your job is to plan, not execute", which the LLM translates to "我目前是 PLAN 模式" in its response. Users reading this think the CLI is stuck in `plan` interaction mode, which is a different thing from the planner sub-agent.

**Problem 2 (F2)**: No structured output format, so plans vary from neat bullet lists to rambling essays depending on the model's mood.

**Design**:

Rewrite the planner prompt in `agents/configs.py` to:
1. Refer to itself as "the planner agent" (a role), never as "plan mode" (which is a user-facing interaction mode).
2. Enforce a structured markdown format.
3. Explicitly tell the LLM that the user is in apply/fix mode and that the plan will be executed by a subsequent worker agent.

**Proposed prompt** (excerpt):
```
You are the planner agent. Your role is to analyze a user request and
produce a structured implementation plan that the worker agent will
execute next. You are NOT the interaction mode — the user is in apply
mode and expects their request to be implemented. Your job is to make
the implementation safe and predictable by planning first.

Output format (markdown, strict):

## Plan

**Goal**: <one-line restatement of what the user wants>

**Scope**: <N files | M steps | risk: low|medium|high>

**Steps**:
1. [verb] <target> — <one-line description>
   Files: `path/to/file.gd`
2. ...

**Risks**:
- <if any, else write "None identified">

**Validation**:
- <how success will be verified>

Do not refer to yourself as being "in plan mode". You are the planner
agent. The worker agent will handle execution next.
```

**Test**: string assertions on `AGENT_CONFIGS["planner"].prompt` — must contain "planner agent", must NOT contain "plan mode" or "PLAN mode" (case-insensitive).

**Separate risk**: Changing a prompt can change model behavior in unpredictable ways. Mitigation: before committing, run the new prompt against a canonical test case manually and compare output structure to the old prompt. Acceptance criteria: plan is still usable, just without the "PLAN mode" hallucination.

## 7. New event taxonomy

Summary table of all new event kinds. `EngineEvent` dataclass stays unchanged — we just use new `kind` strings.

| Kind | When | Data | Display handler |
|---|---|---|---|
| `tool_result_truncated` | context_manager truncates tool output | `{tool_name, char_count, original_length}` | Dim activity line: `tool: <name> result truncated (<N> chars cut)` |
| `session_autosaved` | After successful autosave | `{path}` | Dim activity line: `saved: <file>` |
| `session_autosave_failed` | Autosave exception | `{error}` | Red activity line + console error |
| `diff_failed` | Diff read exception | `{path, reason}` | Activity line: `diff: failed for <path>: <reason>` |
| `version_check_offline` | Update check network error | `{}` | Dim activity line: `update check: offline` |
| `turn_cancelled` | After rollback_current_turn completes | `{}` | Activity line: `turn cancelled` |
| `planner_started` | (v0.9.2, already handled) Before planner pass | `{}` | Rule separator + activity line |
| `planner_finished` | (v0.9.2, already handled) After planner pass | `{used_tools}` | Rule separator + activity line |

All handlers are additive — unknown event kinds continue to be silently ignored by display (existing behavior).

## 8. Test strategy

### Red-green TDD for every issue

Every one of the 28 issues gets a failing test committed first, then the production fix to make it pass. Order per commit:
1. Write failing test
2. Run test — verify it fails for the right reason
3. Write production fix
4. Run test — verify it passes
5. Run full suite — verify no regressions
6. Commit (test + fix together)

### Coverage targets

- **Unit**: every new event kind has an emission test and a display handler test
- **Integration**: cancellation flow has an end-to-end test using a mock LLM that blocks
- **Snapshot**: welcome panel, help table, workspace snapshot on terminal widths 80, 100, 120

### Manual smoke tests before tagging

Before creating the `v1.0.0` tag, manually verify on a real terminal:
1. Fresh `god-code chat` → see spinner during initial reasoning, not empty panel
2. Long prompt that hits `max_tokens` cap → verify bump to 65K resolved truncation
3. Mid-turn Ctrl+C → verify "Cancelled" appears within 2s and next turn starts clean
4. `god-code setup` with existing config → verify overwrite prompt
5. `/resume <bogus_id>` → verify error is actionable, not generic
6. Planner pass in apply mode → verify Rule separators and "planner" language
7. Backend-mode streaming → verify token count > 0 at turn end
8. `/help` → verify sections exist, tab-completion hint present

Each manual test gets a checkbox in the v1.0.0 CHANGELOG entry.

## 9. Migration & compatibility

### Backward compatibility guarantees

- **Config**: All new config fields have defaults. Existing configs work unchanged. Changed default for `max_tokens` (16384 → 65536) is documented prominently.
- **Sessions**: Session file format unchanged. Old sessions load in v1.0.0.
- **API contracts**: Tool contracts, event kinds (old ones), CLI commands all preserved.
- **Backend API**: No changes required in `god-code-api`.
- **Scripts calling god-code**: Non-interactive flows (`/auto`, scripted `ask` commands) unchanged.

### Breaking changes

**None** in external API. Internally, the `on_stream_start` callback may be called with additional context, but the signature stays as `Callable[[], None]` — the new context is passed via side channels (engine attributes).

### Rollback path

If v1.0.0 has a critical regression, users can downgrade to v0.9.2 via `pipx install --force god-code==0.9.2`. Backend `LATEST_VERSION` can be reverted to `0.9.2` to silence update prompts. No data migration needed in either direction.

### Why this commit order

1. Quickest user wins first (B1, B5, A1, A3) — tiny commits with big feel
2. Intermediate layer (A2, B4, C1, G2) — localized fixes, no architectural concerns
3. Higher-touch fixes (commands.py, engine methods) 
4. Riskiest commit (C2 cancellation) last, so if it has issues, the release can ship without it (punt to v1.0.1) while keeping the other 11 improvements

## 11. Risk assessment

### High risk

**C2 — Cancellation architecture**
- Surface: async lifecycle, signal handling, httpx stream cleanup, subprocess management
- Failure modes: deadlock, zombie tasks, orphaned subprocesses, messages appended after "cancel"
- Mitigation:
  - Extensive mocking in unit tests (fake LLM that blocks, fake tool that sleeps, fake subprocess)
  - Manual smoke test on real terminal before tagging
  - Feature-flag escape hatch: `engine._cancellation_enabled = True` default but `False` via env var `GODOT_AGENT_LEGACY_CANCEL=1` if a user hits a crash
  - Ships as the last commit so it can be reverted without affecting other fixes

### Medium risk

**F1, F2 — Planner prompt rewrite**
- Surface: LLM behavior is unpredictable across prompt changes
- Failure modes: plans regress in quality (fuzzier scope, missed risks), format still varies
- Mitigation:
  - Manual A/B: run old and new prompt on 3 canonical test requests, verify new output is at least as good
  - Prompt assertion tests catch obvious regressions (wrong self-description)
  - Rollback is trivial — prompt is a single string constant

**B1 — max_tokens default change**
- Surface: user token budgets
- Failure modes: unexpectedly higher bills for users on metered models
- Mitigation:
  - CHANGELOG prominently documents the change with estimated cost impact
  - Budget warning system (E5 is deferred, but existing budget logic still fires at threshold)
  - Users can revert via `/set max_tokens 16384` if they prefer the old behavior

### Low risk

All display changes, all event additions, all error message improvements — these are additive and isolated. Worst case: a display handler has a typo, a test catches it.

**C1, C3, C4, D1, D2** — transactional fixes — each is a small reordering or added validation. Isolated, testable, reversible.

## 13. Open questions

1. **Cancellation of in-flight tool subprocesses**: should v1.0.0 include subprocess termination for the 3 slow tools (godot_cli, screenshot, image_gen), or defer that to v1.0.1? Current design is to defer — asyncio Task cancellation handles the Python side, subprocesses can finish in detached state with a warning. User input needed.

2. **Planner prompt language**: the prompt rewrite removes "PLAN mode" wording but keeps the general "plan first, don't modify" constraint. Should we also tune the plan format for brevity (5-step cap?) or leave it free-form within the template? Current design: free-form within template. User input welcome.

3. **max_tokens default (65536 vs 32768 vs 128K)**: 65K is the proposed middle ground. 128K (gpt-5.4's full output) is ideal for verbosity but could surprise users with bills. 32K is safer but still truncates big reports. User preference?

4. **Welcome panel fields (D3)**: proposed trim to `{Session, Mode, Model, Project}`. Alternative: `{Session, Mode, Provider/Model, Project}` or `{Mode, Model, Project}` (drop session ID from welcome, keep in `/workspace`). User preference?

5. **Commits 1-11 as one PR or straight-to-main?**: the repo convention (from recent history) is straight-to-main with logical commits. Confirming that's the intended workflow for this release rather than a feature branch + PR.

6. **Should this release include a migration note in CHANGELOG for users on <v0.7 (PyPI-install users who were stuck at 0.6.1 until v0.9.2 released)?** They're jumping ~4 minor versions in one upgrade. Probably yes — link to a brief "what's new since v0.6" digest.

7. **v1.0.0 as GA or RC first?** We could ship `v1.0.0-rc1` to PyPI for a dogfooding pass, then `v1.0.0` GA after a day. Faster path: ship `v1.0.0` directly and issue `v1.0.1` quickly if issues arise. Preference?
