# God Code Pre-launch Security Audit — Design

**Date**: 2026-04-07
**Status**: Approved, ready for implementation plan
**Owner**: 888wing
**Scope**: god-code, god-code-api, god-code-site
**Target**: public launch readiness

---

## Background

`god-code` is preparing for a public launch (PyPI package already at 0.8.1, 3 repos on GitHub, backend deployed to Cloudflare Workers, landing page on Cloudflare Pages). Before driving traffic to the launch, we want a focused pre-launch security audit to ensure no sensitive data leaks after release.

This design is the result of a brainstorming session that clarified two misconceptions and narrowed scope:

1. **Misconception**: An earlier worry that `god-code` repo "stores config.json with API keys" turned out to be a misreading. The `.codetape/config.json` in the repo is Codetape trace metadata (no secrets). Real user API keys are stored in `~/.config/god-code/config.json` under the user's home directory, **outside any repo**, auto-generated by the setup wizard on first run. `.gitignore` is not needed because the file is not inside the repo tree.

2. **Deferred work**: CLI local storage hardening (Keychain), OAuth device flow, and website self-serve key issuance are all valuable but deferred to post-launch. They are not blockers for release.

## Goals

- Complete a focused security audit of all 3 repos + published artifacts before launch.
- Remediate any findings in-place (revoke + fix + document).
- Produce baseline security/privacy documentation that public launch requires (`SECURITY.md`, `PRIVACY.md`, `DEPLOYMENT.md`).
- Install local-only prevention (`.gitleaks.toml` + pre-commit hook) to catch regressions.

## Non-goals

The following are explicitly **out of scope** for this pre-launch audit. They are valuable but deferred to post-launch hardening sprints:

- Upgrading CLI local storage from plain JSON to OS Keychain (macOS Keychain / libsecret / DPAPI).
- OAuth device flow between CLI and godcode.dev.
- Website self-serve key issuance flow (signup → dashboard → create key).
- GitHub Actions CI integration of `gitleaks` (post-launch).
- Dependency vulnerability scanning (`pip-audit`, `npm audit`).
- Rate limiting on `/v1/admin/keys` (only fix if audit discovers an obvious issue).
- Responsible disclosure program beyond a basic email contact.

## Threat model

This audit defends against two threat levels:

- **Level A — Accidental disclosure**: secrets committed to git, leaked via screenshots, public gists, or exposed via misconfigured infrastructure.
- **Level B — Shared machine / curious coworker**: other user accounts or people with shell access to the same machine reading plaintext files in the user's home directory.

The audit does **not** defend against:

- Local malware running under the user's own session (Level C).
- State-actor threats or targeted attacks.
- Supply-chain attacks on upstream dependencies.

## Scope: three tracks

### Track A — Repo and published artifact secrets

Audit the 3 repos (god-code, god-code-api, god-code-site) and the published PyPI wheel for accidentally committed secrets.

**Checks**:

1. Run `gitleaks detect --source . --log-opts="--all"` against full git history on each repo.
2. Download the current PyPI wheel (`pip download god-code==0.8.1 --no-deps`), unzip it, and verify no sensitive files are packaged (config.json, auth.json, .env, .key, .agent_sessions, .pypirc).
3. Amend `pyproject.toml` to add an explicit `[tool.hatch.build.targets.wheel].exclude` list, preventing future accidental packaging.
4. Check for `.env.example` or similar template files in each repo and verify their content is safe to publish.
5. Grep `tests/fixtures/` in all 3 repos for hardcoded credentials.

### Track B — Deployment secrets

Verify that `god-code-api` and `god-code-site` deployment secrets are injected via platform-native secret management, not plaintext config files.

**Checks**:

1. `wrangler secret list` on `god-code-api` to verify `ADMIN_SECRET` and provider API keys (OpenAI, Anthropic, Gemini, xAI) are present as secrets, not in `wrangler.toml` or source.
2. Review Cloudflare Workers logs in the dashboard to confirm error paths do not print request headers (which would leak `Authorization: Bearer gc_live_*`).
3. Verify `god-code-site` Cloudflare Pages build env vars contain no secrets.
4. Write a `docs/DEPLOYMENT.md` for both `god-code-api` and `god-code-site` documenting the full secret inventory, how each is set, and how to rotate.

### Track C — User data handling

Audit every place where user-originated data is stored or transmitted, and publish a privacy document.

**Checks**:

1. Produce a data-flow table (to be embedded in `PRIVACY.md`) listing every data point that flows from CLI → god-code-api → LLM provider:
   - **CLI uploads**: Godot project files (when tools read them), screenshots (when vision tools capture them), conversation history, session metadata.
   - **API persists**: `usage_log` fields (agent_role, provider, model, prompt_tokens, completion_tokens, cost_estimate, quality_score) — confirm prompt/completion content is **not** stored.
   - **KV stores**: `waitlist:{email}` entries (email, joined_at, source/referer).
2. Grep `god-code` codebase to confirm `autosave_session` writes only to local disk (`.agent_sessions/`) and never uploads session content to the backend.
3. Decide whether to remove the `referer` field from waitlist KV entries (may contain tracking params / PII).
4. Decide waitlist retention policy and implement cleanup mechanism.

## Decisions made during brainstorming

The following decisions were made during the brainstorming session and are authoritative:

| Decision | Value |
|---|---|
| Waitlist retention period | Until launch ends (then purge) |
| Security contact email | `info@do-va.com` |
| History rewrite if `gitleaks` finds historical secrets | **No force-push.** Revoke the leaked credential only. Accept that the secret remains in git history (revoked and useless). |
| Post-fix re-audit | Not required. Fix in-place and ship. |

The rationale for the "revoke only, no history rewrite" decision: force-pushing rewrites is disruptive (breaks forks, CI, local clones) and its value is limited once the secret is revoked. A revoked secret in git history is a defunct string, not an active risk. The cost of disruption outweighs the marginal benefit.

## Remediation strategy

Remediation is **in-place and incremental**:

- **Secret in git history (revoked only)**: identify the leaked credential, revoke it immediately at the provider, add a note in `docs/SECURITY-HISTORY.md` documenting the revocation, do **not** rewrite history.
- **Secret in working tree**: revoke, remove from file, add to `.gitignore`, commit.
- **PyPI wheel includes sensitive file**: fix hatch `exclude` list, bump patch version (0.8.1 → 0.8.2), republish to PyPI.
- **Workers logs contain PII**: add log-redaction middleware in `god-code-api`, redeploy.
- **Waitlist KV has PII risk**: modify handler to drop `referer` field, redeploy, purge existing entries as part of the decision.

## Documentation deliverables

### SECURITY.md (3 repos, similar content, repo-specific tweaks)

Structure:

- **Supported versions**: `god-code >=0.8.0`; `god-code-api` / `god-code-site` always latest deployed.
- **Reporting a vulnerability**: email `info@do-va.com`; expected response time **5 business days**.
- **Out of scope**: user-provided BYOK provider vulnerabilities (OpenAI, Anthropic, etc.), issues in user's own Godot projects, social engineering.
- **Disclosure policy**: request reporters hold disclosure until fix is released.

### PRIVACY.md (god-code + god-code-site)

Structure:

- **Data flow table** (CLI → API → Provider): every data point, where it goes, where it persists.
- **What we store**: `usage_log` fields (no content), waitlist emails (with retention period), local session files (on user's machine only).
- **What we do NOT store**: prompt content, completion content, user's Godot project code on our servers (CLI sends to LLM provider directly in BYOK mode).
- **User rights**: how to request deletion (email `info@do-va.com`), how to query stored data.
- **Third parties**: LLM providers the user has selected (their privacy policies apply to BYOK traffic).
- **Basic GDPR / CCPA alignment**: lawful basis, data minimization, retention.

### DEPLOYMENT.md (god-code-api + god-code-site)

Structure:

- **Secret inventory**: every secret binding, its purpose, where it lives (CF Workers secret / CF Pages build env).
- **How to set**: `wrangler secret put SECRET_NAME`, CF dashboard steps.
- **How to rotate**: step-by-step runbook including redeployment.
- **Emergency revocation**: what to do if a secret leaks.

### Audit report (internal, not published)

`docs/plans/2026-04-07-god-code-prelaunch-security-audit-report.md` — the final report listing each check, result, and remediation action taken. This stays in the god-code repo for internal reference.

## Success criteria

The audit is complete and launch-ready when all of the following are true:

- `gitleaks` full-history scan on all 3 repos produces zero actionable findings (or all findings have been remediated via revoke + document).
- PyPI wheel 0.8.1 has been downloaded and inspected; no sensitive files found inside.
- `pyproject.toml` has an explicit `exclude` list in the hatch build config, and a local test build (`uv build` or `python -m build`) produces a clean wheel.
- `wrangler secret list` on `god-code-api` confirms `ADMIN_SECRET` and all provider keys are set as secrets.
- 3 × `SECURITY.md` published (god-code, god-code-api, god-code-site).
- 2 × `PRIVACY.md` published (god-code, god-code-site) with the data-flow table filled in.
- 2 × `DEPLOYMENT.md` published (god-code-api, god-code-site).
- Waitlist KV entries handled per decision (referer field removed if we decide to remove it; retention policy documented).
- `.gitleaks.toml` + pre-commit hook installed locally in all 3 repos.

## Out-of-band notes

- If `gitleaks` finds historical secrets, the "revoke only, no history rewrite" policy means a secret revoked at the provider is considered remediated. Document the incident in an internal `SECURITY-HISTORY.md` so future audits have context.
- The Cloudflare Workers log review (Track B, check 2) is a manual dashboard action and should be captured as a screenshot or text log in the audit report.
- PyPI wheel inspection should be done against the live artifact (downloaded from PyPI), not a local `dist/` build, because the local build may not be what users actually install.
